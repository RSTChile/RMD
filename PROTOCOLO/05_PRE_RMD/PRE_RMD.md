# PRE-RMD — FASE DE PREPARACIÓN DEL PROTOCOLO

## Naturaleza

Esta fase conecta el análisis cualitativo (MiE) con la ejecución cuantitativa del Protocolo RMD 2.0.

No es una fase interpretativa.

Es una fase estructural de preparación del sistema de medición.

---

## Objetivo

Asegurar que:

- el modelo tiene todos los insumos necesarios
- las matrices están disponibles
- los vacíos están identificados
- la medición es posible

---

## Regla central

No se permite iniciar el Protocolo sin completar esta fase.

Si hay vacíos críticos:

→ el análisis se detiene

---

## Componentes obligatorios

### 1. Verificación de matrices

Se debe verificar:

- existencia de matrices necesarias
- acceso a archivos
- correspondencia con variables del modelo

---

### 2. Mapa de variables disponibles

Identificar:

- variables que se pueden medir
- variables que no tienen datos

---

### 3. Manifiesto de ejecución

Listado de:

- vacíos críticos
- riesgos de medición
- oportunidades de exaptación

---

### 4. Mapa de comodines (LF)

Identificar:

- dónde aplicar “No sé”
- dónde puede aparecer “Discrepo”
- dónde abrir escenarios “¿Y si…?”

---

## Rol de la IA (LF activa)

La IA actúa como analista en:

- detección de vacíos estructurales
- anticipación de conflictos de medición
- identificación de oportunidades de exaptación

---

## Qué está permitido

- revisar matrices
- identificar faltantes
- anticipar tensiones del modelo
- preparar el sistema de cálculo

---

## Qué está prohibido

- comenzar medición sin completar esta fase
- inventar datos faltantes
- ignorar vacíos críticos
- mezclar matrices

---

## Formato de salida

### 1. Estado del sistema

Matrices disponibles / faltantes.

---

### 2. Variables medibles

Listado claro de variables que pueden calcularse.

---

### 3. Vacíos críticos

Variables o datos inexistentes.

---

### 4. Manifiesto de ejecución

- riesgos
- oportunidades
- advertencias

---

### 5. Mapa de comodines

- No sé → dónde se activa  
- Discrepo → posibles tensiones  
- ¿Y si…? → escenarios críticos  

---

## Uso de LF

Mandatorio:

→ “No sé” para vacíos reales  
→ “Discrepo” ante incoherencias  
→ “¿Y si…?” para anticipar escenarios  

---

## Prohibición estructural

No se permite:

- avanzar sin este diagnóstico
- ocultar vacíos
- simular completitud del modelo

---

## Iteración obligatoria

Después de esta fase:

- validar con el analista humano
- completar datos faltantes si es posible
- ajustar el sistema

---

## Resultado esperado

Un sistema listo para ejecutar el Protocolo RMD con:

- trazabilidad completa
- variables identificadas
- riesgos explícitos

Condición de entrada al Protocolo.

---

## Acceso ampliado a Variables y Métricas (CRÍTICO)

Además de las variables definidas en cada subcategoría del Protocolo, el sistema dispone de un repositorio completo de Variables y Métricas.

Esto implica:

- Las subcategorías definen el flujo base
- El repositorio completo permite expansión analítica

---

## Regla de uso

La IA puede:

- incorporar variables adicionales no incluidas en la subcategoría
- complementar el análisis cuando el caso lo requiera
- profundizar dimensiones no capturadas por el flujo estándar

---

## Condiciones obligatorias

Toda variable adicional debe:

- provenir del repositorio oficial de Variables y Métricas
- estar justificada en relación al caso
- mantener trazabilidad (fuente y motivo de uso)

---

## Uso de LF (expansión controlada)

Esta capacidad corresponde a:

→ aplicación de Libertad Funcional (LF)

Permite:

- ampliar el análisis
- detectar dimensiones ocultas
- aplicar exaptación

---

## Prohibición estructural

No se permite:

- inventar variables
- usar variables sin trazabilidad
- romper coherencia del modelo

---

## Resultado esperado

Un sistema que:

- sigue el protocolo base
- pero puede expandirse de forma controlada
- manteniendo rigor y trazabilidad
